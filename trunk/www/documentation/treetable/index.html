Introduction

The most common use of a tree table component is to display a hierarchical list of objects. In this usage, each row represents an object, and each column represents some attribute of that object. Additionally, an object may have one or more "child" objects which are shown indented beneath the "parent" object.

Note: "object" means an object in the mind of the user (file, appointment, breakpoint, etc), not a java.lang.Object.

This document is a user interface specification for that usage of a TreeTable in JDNC (called a Hierarchical Object List in this document). It contains sections for all the behaviors and the appearance details of that usage, with sub-sections for four different looks and feels (JLF/Ocean, Windows XP, GNOME 2.x and Macintosh OS X 10.3).

Note: JDNC will not deliver a Macintosh look and feel. This is documented here to help demonstrate the breadth of features the component must be able to support, and to provide guidance to anyone that would like to provide this support.

This specification covers two categories of details. The first is information that must be provided by the component itself. The second category is information that an application developer can use to make their application more professional and to avoid "re-inventing the wheel" for these common design decisions.

Note that after the Overview section, the terms MUST, SHOULD and MAY in this document have the formal meaning found in http://rfc.net/rfc2119.html.

Much of the information about what is standard on the various platforms comes from some demo applications. These are provided for GNOME/GTK+2.0, Macintosh.

List Behaviors

Row Selection

A list must allow 0 or 1, or multiple rows to be selected. Continuous and discontinuous sets of rows may be selected.

When a row is selected, the whole row must be selected.

One term will be useful to understand the rules below:

  • Anchor row: a row that the underlying software uses to determine where to make continuous selections from. The user is never given any feedback to indicate which row is the anchor.

Details For

Other notes:

  • The keyboard focus indicator's visibility is controlled on a window basis. The general rules are:
    • It is not shown when a window is first displayed or when it is brought to the foreground (nor is it shown when the window is in the background).
    • Any time a keydown event occurs in the window, the focus indicator will be shown (if appropriate) on the currently focused component.
      • Certain keys, however, have an immediate effect, and that action more or less captures the keydown so that the window doesn't get it and thus doesn't show the keyboard focus indicator (e.g. Alt, certain F-keys)
  • When an arrow key is pressed, the focus marker must be moved to the cell above (up-arrow) or below (down-arrow) the cell that had the keyboard focus marker before the key was pressed.
  • The Home key must behave just like using the up arrow key, in the table above, except that instead of moving one row up, it moves to the first row (and the first column) in the list. Thus, unmodified Home must select the first row in the list and deselects all others, Ctrl-Home must move the keyboard focus to the first row and changes no selection, etc.
  • The End key must be like the Home key except it affects the last row (and last column) in the list.
  • The PageUp key must behave just like using the up arrow key, except that it moves the selection and focus to the first whole row (unchanged column) visible on the screen, and if it is already on that row the view must automatically scroll to the topmost visible row (unchanged column) on the next view-full of information (in that case, the row that was at the top should be kept as the last whole visible row in the scrolled view).
  • The PageDown key must be like the PageUp key except it affects the last visible row (unchanged column)in the list.
  • If the focus indicator is in the first row in the list then pressing the up arrow, home, or PageUp keys must have no effect. If it is in the last row, then down arrow, End and PageDown must have no effect.
  • If the rows are not draggable, the left drag must always be treated like a click.
  • If the list allows only one row to be selected at a time, then the shift and ctrl modifiers must be ignored and all behavior should be like an unmodified click, arrow key, or space.
  • Applications must ignore modifier keys on a right click.
  • Applications may provide a "Select All" command (generally Ctrl-A) which selects all the rows in the list (but doesn't affect the anchor)
  • Applications must assure that a right click will have the same effect as an unmodified left click excpet that a contextual menu must be shown on mouseUp and any other mouseUp behavior (e.g. deselection) is not done.

Ordinarily the component provides a keyboard focus in the individual cells. This allows the component to provide a way to manipulate the size and ordering of the columns from the keyboard in a cross-platform manner. However, the component should provide a mode where the keyboard focus spans all cells of a row, which is more consistent with the Windows style. In this case, however, it the columns must not be able to be resized or repositioned.

The rules for keyboard navigation and row selection presented here are derived from those in the System Monitor tree view (type Ctrl-Alt-Del from the GNOME environment to get it). This seems to be basically consistent in its behavior with other similar components in the envrionment (e.g. the list view in Nautilus).

There is some relevant material in the GNOME standards for keyboard accessibility.

The rules are identical to the Windows ones, listed above, with these exceptions:

  • Ctrl-Shift Left Click must be identical to Ctrl Left Click
  • Ctrl-Shift Up/Down Arrow must be identical to Ctrl Up/Down Arrow
  • Shift Space must be identical to Space
  • Ctrl-Shift Space must select nothing, deselect nothing, and not affect the anchor
  • The algorithm for determining what PageUp and PageDown are different than on Windows. On GNOME, if N whole rows can be shown in the list, then pressing PageDown must affect the Nth row down the list (including counting the one that started with the focus). PageUp counts the same number of rows up. (Put another way, pressing PageUp or PageDown on GNOME always skips a fixed number of rows, while on Windows the number of rows skipped will vary based on which row started out with the keyboard focus).
  • Keyboard focus must be able to be put in the column headers
  • If the keyboard focus indicator was showing in the component when it last had focus, and it is again given focus, the indicator must be visible, no matter how the component got focus.
  • However, the focus indicator in rows must be hidden as a result of mouse operations in the component, and must be shown after keyboard gestures. [Note: This may be a bug in GNOME, since other components show the keyboard focus after a mouse gesture]

Column Movement

The user should be able to rearrange the columns in the list.


Before moving Value
After moving Value

To move a column with the mouse, the user must press anywhere in the column header and then drag the column either left or right and then drop it wherever she wishes. Doing this never changes any other aspect of the list.

If Esc is pressed while dragging, the drag must be cancelled.

To move a column with the keyboard, the application must provide Move Left and Move Right commands in the menus. The column which contains the keyboard focus marker must move to the right or left by one column (and the keyboard focus must move with it). If a column is leftmost or rightmost then the menu item(s) must be disabled.

It would be much nicer if the component would provide a mode that a single "Move Column" command could invoke. This would start a mode where the column could be moved with repeated presses of the arrow keys, and any other key than Esc would exit the mode and leave the column in place. The mouse pointer would also moved to the center of the column header when this mode is started, and returned to its old position when the mode is ended. This would mimic the "Move Window" command found in Windows and GNOME. (nicer because it clutters the menu less, and allows more rapid column movement for keyboard users)

Note: There is no standard way to move columns with the keyboard on on any of these platforms. Several applications have dialog boxes which allow the user to change the column order (and sometimes the size) from there. The method presented here is "better" since it can coexist with these other schemes, and can be done uniformly across all platforms with a minimum of interface overhead.

When the keyboard focus spans the whole row, then the columns must not be movable by the user.

Dragging Visual Feedback

The visual feedback when dragging a column on Windows is to only move a transparent version of the header, and to show some drop-target feedback as this is moved. The transparent version of the header is positioned under the pointer so that the point the user first pressed the mouse on remains under the pointer's hot spot.

There is no autoscrolling, but a column may be dropped "offscreen" if the drop point is offscreen.

Application Details

The application should remember the column order in both primary and secondary windows across restarts of the program so that the user is not forced to rearrange the columns every time.

An application may have a dialog box (accessible from the View menu) which would allow the user to adjust which columns are shown in the component.

Column Resizing

The user should be allowed to change the width of columns (optionally with a minimum and/or maximum size).


Before Resizing Value
After Resizing Value

To resize with the mouse, the mouse pointer must be pressed in the region between the column headers, and then dragged. The exact result of this drag operation will depend on the resize mode for the whole list.

To resize with the keyboard, the Make Narrower and Make Wider menu commands must be used. These apply to the column that contains the keyboard focus. These commands change the width by 3 pixels.

As with column movement, it would be best if the component provided a "Resize Column" mode which would operate similar to the Resize command on Windows and GNOME. When invoked via a menu command, left and right arrow keys would adjust the right edge of the current column, any key other than arrows and Esc would leave the mode and leave the column at its new width, while Esc would return the column to the original size.
An alternate (and better-still) strategy would be that when the mode is first invoked, the arrow keys would first be used to choose a column to adjust, and then once that mode was ended a mode to adjust the selected column would start. This has the benefit of not forcing some of the unusual per-cell keyboard focus feedback that is required elsewhere in this spec.

As with column movement, there is no standard way to adjust column widths with the keyboard on any of the platforms. This solution doesn't conflict with any conventions and so can be applied uniformly across looks and feels.

Details For

Resizing Edge

Resizing is always done on the rightmost edge of a column. The left edge of the leftmost column can not be resized.

Resizing Pointer

Standard left/right resize cursor is shown when it enters the region between the column headers, and during the drag.

Resize Feedback

Columns are resized "live". If the pointer is moved outside of the component, the column will continue to be resized, even to the full width of the screen(s).

Resizing Regions

The region between the column headers is a 14 pixel region equally spanning the edges of two columns (the column divider appears fully in the leftmost of the two columns).

Column Sorting

General

A list may allow its rows to be sorted by the values in one of its columns.


Before Sorting "Name" Ascending
After Sorting "Name" Ascending

If there are child objects in the list, then they must be sorted within the scope of their parent. That is, they are always kept under their parent, and are sorted relative only to one another. Alternately, the list may be "flattened" (that is, the hierarchy must be removed), in which case all objects would be sorted as siblings.

The sort should be done on the visible content of the column (e.g. an alphabetic or numeric sort). However, in some cases it may be done on an underlying value. For example in a list of bugs, the "Severity" column might contain values "Neutral", "Bad", and "Very bad", and these should be sorted in that order rather than an alphabetic order (Bad, Neutral, Very bad).

Whenever a different column is made the sort column, the sort should be done as a stable sort (this keeps items in the old column in the same relative order when there are identically valued items in the new column). This may be ignored, however, if the list is switched to an "unsorted" state.

Most lists that allow sorting should require one column to be sorted at all times. In this case, the most appropriate column should be marked as sorted in the most appropriate direction by default. On rare occasion, however, a list may allow an "unsorted" or "natural order" state (this is only done for responsiveness reasons in lists that are updated very frequently and where a user-imposed sort is not crucial). In this case, the list may appear in the unsorted state by default.

On very rare occasion, a set of users may have a need to sort on multiple columns simultaneously. This specification doesn't discuss how that should be done.

Details For

Sorting With a Mouse
  • The user must click on the column header she wishes to sort the objects by.
    • If the column wasn't already the sorted column, then the column must be sorted ascending
    • If the column was already the sorted column, then the sort order must be toggled (ascending to descending, or vice versa)
      • If the list allows an "unsorted" state, then the list must cycle from ascending, to descending, to switching the whole list to an unsorted state). [Note: I know of no Windows apps which do this. However, this has proven to be a reasonable paradigm in the NetBeans envrionment]
Sorting With a Keyboard
  • Lists should be able to be sorted with the "Sort By" menu (see below) which is provided by the application. This should be in the contextual menu for the list and may be in the View menu. [Note: This is not at all "standard" on Windows. It is specified here as a Sun-defined enhancement to provide a consistent cross-application/cross-platform way to accomplish this].
  • Some applications may provide other means to accomplish sorting from the keyboard. For example, Windows Explorer has a a dialog box available from the View menu may allow a variety of details to be specified about the list.
